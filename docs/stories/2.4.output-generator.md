# Story 2.4: Output-Generator Primitive (Template-Based Output)

## Status

Review

## Story

**As a** Python developer using Beddel,
**I want** an `output-generator` primitive that extracts and returns the resolved template from the step config,
**so that** I can define workflow steps that produce structured or plain-text outputs from previously resolved variable references without requiring an LLM call.

## Acceptance Criteria

1. Register an `output-generator` async primitive function that accepts `(config: dict, context: ExecutionContext)` and returns the resolved template value
2. Extract `config["template"]` and return it as-is — the template value has already been resolved by the VariableResolver before reaching the primitive
3. Support dict template values (e.g., `{"topics": "...", "summary": "..."}`) — return the dict directly
4. Support string template values (e.g., `"Final report: ..."`) — return the string directly
5. Raise `PrimitiveError` with `BEDDEL-EXEC-001` when the `template` key is missing from config
6. Log template extraction at DEBUG level via structured logging
7. Unit tests achieve >80% coverage for `primitives/output.py`

- **Logger**: Use `logging.getLogger("beddel.primitives.output")`.

### Output-Generator vs LLM/Chat Primitives — Key Difference

The `llm` and `chat` primitives call an external LLM provider and return `LLMResponse` objects. The `output-generator` is a **pure data extraction primitive** — it takes the already-resolved config, extracts the `template` field, and returns it directly. No LLM call, no provider, no network I/O.

The VariableResolver has already substituted all `$stepResult.*` and `$input.*` references before the primitive receives the config. The output-generator's only job is to extract and return `config["template"]`.

### YAML Config Shape (from fixtures)

```yaml
- id: "format"
  type: "output-generator"
  config:
    template:
      topics: "$stepResult.analysis.text"
      summary: "$stepResult.summary.text"
      pipeline:
        steps: 3
        status: "completed"
  result: "report"
```

After variable resolution, config arrives as:
```python
{
    "template": {
        "topics": "AI, Machine Learning, NLP",
        "summary": "This report covers three key areas...",
        "pipeline": {
            "steps": 3,
            "status": "completed",
        },
    }
}
```

The primitive returns the `template` dict as-is. The executor stores it under the `result` key (`"report"`) in `step_results`.

String templates are also supported:
```yaml
- id: "greeting"
  type: "output-generator"
  config:
    template: "Hello, $input.name! Your analysis is complete."
  result: "message"
```

After resolution: `config = {"template": "Hello, Alice! Your analysis is complete."}` → returns the string.

### Data Models

The output-generator uses minimal models from `beddel.domain.models`:

`ExecutionContext`:
- `workflow_id: str` — Workflow identifier
- `input: dict[str, Any]` — Workflow input data
- `step_results: dict[str, Any]` — Results from previous steps
- `env: dict[str, str]` — Environment variables
- `metadata: dict[str, Any]` — Runtime metadata (e.g., `llm_provider`)

`PrimitiveError(ExecutionError)` — for primitive-specific failures, uses `ErrorCode.EXEC_STEP_FAILED` (BEDDEL-EXEC-001)

Note: `LLMRequest`, `LLMResponse`, `Message`, `ProviderError` are NOT needed — this primitive does not interact with any LLM provider.

### Implementation Reference

The full implementation is straightforward (~20 lines):

```python
async def output_primitive(config: dict[str, Any], context: ExecutionContext) -> Any:
    template = config.get("template")
    if template is None and "template" not in config:
        raise PrimitiveError(
            "output-generator requires a 'template' key in config",
            code=ErrorCode.EXEC_STEP_FAILED,
            details={"primitive": "output-generator", "hint": "Add template field to config"},
        )
    logger.debug("Output template extracted: type=%s", type(template).__name__)
    return template
```

Important edge case: `config.get("template")` returns `None` both when the key is missing AND when the value is explicitly `None`. Use `"template" not in config` to distinguish. If the key exists with value `None`, return `None` (valid). If the key is missing entirely, raise `PrimitiveError`.
