# Story 6.1: Streaming-First Execution (Event-Driven)

## Status

InProgress

## Story

**As a** Python developer building real-time applications with Beddel,
**I want** the `WorkflowExecutor` to expose an `execute_stream()` method that yields structured `BeddelEvent` objects as an `AsyncGenerator`, and the SSE adapter to consume this event stream directly via a new `stream_events()` method,
**so that** I can observe workflow execution in real time with typed lifecycle events (start, end, chunk, error) without waiting for the full result, while the existing `execute()` method remains unchanged for backward compatibility.

## Acceptance Criteria

1. Add a `BeddelEventType` enum to `domain/models.py` with values: `WORKFLOW_START`, `WORKFLOW_END`, `STEP_START`, `STEP_END`, `TEXT_CHUNK`, `STEP_RESULT`, `ERROR`
2. Add a `BeddelEvent` Pydantic model to `domain/models.py` with fields: `type` (`BeddelEventType`), `workflow_id` (`str`), `step_id` (`str | None`), `data` (`Any`), `timestamp` (`datetime`)
3. Add `execute_stream(workflow, input_data) -> AsyncGenerator[BeddelEvent, None]` to `WorkflowExecutor` that yields `BeddelEvent` instances at each lifecycle point (workflow start/end, step start/end, text chunks, step results, errors)
4. `execute_stream()` fires lifecycle hooks AND yields events at the same lifecycle points (dual-write) — hooks and events are always in sync
5. `execute_stream()` yields an `ERROR` event and stops iteration when a step fails mid-workflow (no partial results after error)
6. `BeddelEvent.data` never contains raw Python tracebacks — error events carry structured payloads with `code`, `message`, and `details` keys only
7. The existing `execute()` method remains unchanged — all current tests pass without modification
8. Add `stream_events(stream: AsyncGenerator[BeddelEvent, None]) -> AsyncIterator[SSEEvent]` to `BeddelSSEAdapter` that maps each `BeddelEvent` to an `SSEEvent` (event type = lowercase `BeddelEventType` name, data = JSON-serialized `BeddelEvent.data`)
9. Promote `_build_error_event` in `integrations/sse.py` to a public function `build_error_event` (fixes SSE-001 from Story 5.2 QA)
10. Fix `SSEEvent.serialize()` to handle multi-line data by splitting into multiple `data:` lines per SSE spec (fixes SSE-003 from Story 5.2 QA)
11. Refactor `_sse_bridge()` in `integrations/fastapi.py` to call `execute_stream()` directly and pipe through `adapter.stream_events()` — no more `await execute()` then iterate
12. `stream_events()` yields a final `SSEEvent(event="done", data="[DONE]")` sentinel after the `WORKFLOW_END` event
13. `execute_stream()` handles `GeneratorExit` (client disconnect) gracefully — logs a warning and cleans up without raising
14. Unit tests achieve >80% coverage for modified code paths in `executor.py`, `sse.py`, and `fastapi.py`

### Previous Story Insights (from E5.2 — SSE Streaming Response Adapter)

Story 5.2 created the framework-agnostic `BeddelSSEAdapter` with `stream()` and `stream_iterator()` methods. QA flagged three items relevant to this story:
- **SSE-001** (medium): `_build_error_event` is imported as a private function across module boundary in `fastapi.py` — this story promotes it to public `build_error_event` (Task 3)
- **SSE-003** (low): `serialize()` does not split multi-line data into multiple `data:` lines per SSE spec — this story fixes it (Task 3)
- **SSE-004** (low): Strict `AsyncIterator` check may miss some async streams — not addressed in this story (Phase 2 scope)

The current `_sse_bridge()` in `fastapi.py` does `await executor.execute()` then iterates the result — this story replaces that with direct `execute_stream()` consumption, eliminating the await-then-iterate pattern.

[Source: docs/stories/5.2.sse-streaming-adapter.md — QA Results]

### Architecture Context

This story introduces the event-driven execution layer — a new async generator path alongside the existing blocking `execute()`. The architecture flow changes for streaming:

**Before (5.1/5.2):**
```
FastAPI → await execute() → ExecutionResult → BeddelSSEAdapter.stream() → SSEEvent
```

**After (6.1):**
```
FastAPI → execute_stream() → AsyncGenerator[BeddelEvent] → BeddelSSEAdapter.stream_events() → SSEEvent
```

The blocking path (`execute()`) is unchanged. The streaming path now yields structured events at each lifecycle point instead of waiting for the full result.

**Dual-write principle:** Both hooks and events fire at the same lifecycle points. The executor is the single source of truth — hooks are called AND events are yielded in the same code path. This avoids drift between the two observation mechanisms.

[Source: Architect (Winston) — Epic 6 design review]

### BeddelEvent Design

```python
class BeddelEventType(StrEnum):
    WORKFLOW_START = "workflow_start"
    WORKFLOW_END = "workflow_end"
    STEP_START = "step_start"
    STEP_END = "step_end"
    TEXT_CHUNK = "text_chunk"
    STEP_RESULT = "step_result"
    ERROR = "error"

class BeddelEvent(BaseModel):
    type: BeddelEventType
    workflow_id: str
    step_id: str | None = None
    data: Any = None
    timestamp: datetime = Field(default_factory=lambda: datetime.now(UTC))
```

Phase 1 (this story) covers the 7 event types above. Phase 2 (separate story) adds typed tool events, state snapshots, and structured `data` payloads per event type.

[Source: Architect (Winston) — Epic 6 design review]

### Event Sequence Grammar

A valid event stream for a successful N-step workflow follows this grammar:

```
WORKFLOW_START
  (STEP_START (TEXT_CHUNK)* STEP_RESULT STEP_END)*
WORKFLOW_END
```

For an error mid-workflow:

```
WORKFLOW_START
  (STEP_START (TEXT_CHUNK)* STEP_RESULT STEP_END)*
  STEP_START (TEXT_CHUNK)*
  ERROR
```

Key invariants:
- `WORKFLOW_START` is always the first event
- `WORKFLOW_END` is always the last event (success path only)
- Every `STEP_START` has a matching `STEP_END` (except the failing step on error)
- `ERROR` terminates the stream (no `STEP_END` or `WORKFLOW_END` after it)
- `TEXT_CHUNK` events only appear between `STEP_START` and `STEP_RESULT` for streaming steps

### Dual-Write Implementation Strategy

QA (Quinn) recommended extracting a shared execution core to avoid drift between `execute()` and `execute_stream()`. For Phase 1, the pragmatic approach is:

1. `execute_stream()` is a new method with its own step loop (mirrors `execute()` structure)
2. Both methods call the same `_execute_step()` internal method for step execution
3. The difference: `execute_stream()` yields events at each lifecycle point; `execute()` collects results
4. Hooks fire in both paths via the same `for hook in self._hooks` pattern

A shared core refactor (extracting step iteration into a generator consumed by both methods) is deferred to a future hardening story to keep this story focused.

### Error Event Payload Structure

`BeddelEvent.data` for `ERROR` events MUST be a structured dict:

```python
{
    "code": "BEDDEL-EXEC-001",      # ErrorCode string (or "INTERNAL_ERROR")
    "message": "Step 'summarize' failed: ...",  # Human-readable message
    "details": {}                     # Additional context (never tracebacks)
}
```

For `BeddelError` subclasses, extract `code`, `message`, `details` directly. For generic exceptions, use `code="INTERNAL_ERROR"` and `message=str(exc)`. Never include `traceback.format_exc()` or `exc.__traceback__` in the payload.

[Source: QA (Quinn) — Story 6.1 review concerns]

### SSE Multi-Line Data Fix (SSE-003)

The SSE spec requires multi-line data to be split into multiple `data:` lines:

```
# Before (broken for multi-line):
data: line1\nline2

# After (correct per W3C spec):
data: line1
data: line2
```

Implementation: in `SSEEvent.serialize()`, split `self.data` on `\n` and emit each line as a separate `data:` field.

[Source: https://html.spec.whatwg.org/multipage/server-sent-events.html]

### ILifecycleHook Protocol (Current State)

```python
@runtime_checkable
class ILifecycleHook(Protocol):
    async def on_workflow_start(self, workflow: WorkflowDefinition, input_data: dict[str, Any]) -> None: ...
    async def on_step_start(self, step: StepDefinition) -> None: ...
    async def on_step_end(self, step: StepDefinition, result: Any) -> None: ...
    async def on_workflow_end(self, workflow: WorkflowDefinition, result: Any) -> None: ...
    async def on_error(self, error: Exception) -> None: ...
    async def on_llm_start(self, request: LLMRequest) -> None: ...
    async def on_llm_end(self, request: LLMRequest, response: LLMResponse) -> None: ...
```

`execute_stream()` fires the same 5 workflow/step/error hooks as `execute()`. The `on_llm_start`/`on_llm_end` hooks are fired inside primitives (not the executor) and are unaffected by this story.

[Source: src/beddel-py/src/beddel/domain/ports.py]

### Technical Constraints

- Python 3.11+ with async/await [Source: docs/architecture/tech-stack.md]
- `from __future__ import annotations` for deferred evaluation [Source: docs/architecture/coding-standards.md]
- Type hints required on all public functions [Source: docs/architecture/coding-standards.md]
- Structured logging via named loggers with %-style formatting [Source: docs/architecture/coding-standards.md]
- `AsyncGenerator` return type requires `from collections.abc import AsyncGenerator` (not `typing.AsyncGenerator`) [Source: Python 3.11+ best practice]
- `execute()` MUST NOT be modified — backward compatibility is non-negotiable [Source: Architect (Winston)]
- Error events MUST NOT contain raw tracebacks [Source: QA (Quinn)]
- `GeneratorExit` is a `BaseException`, not `Exception` — must be caught explicitly [Source: Python stdlib]
