# Story 2.5: Call-Agent Primitive (Nested Workflow Invocation)

## Status

Review

## Story

**As a** Python developer using Beddel,
**I want** a `call-agent` primitive that invokes another workflow by its agent ID,
**so that** I can compose complex behaviors by delegating to sub-workflows and reusing workflow definitions as building blocks.

## Acceptance Criteria

1. Register a `call-agent` async primitive function that accepts `(config: dict, context: ExecutionContext)` and returns the nested workflow's output
2. Extract `config["agentId"]` to identify the target workflow — raise `PrimitiveError` with `BEDDEL-EXEC-001` when missing
3. Extract `config["input"]` as the input data for the nested workflow (default to empty dict `{}` if not provided)
4. Retrieve a workflow loader callable from `context.metadata["workflow_loader"]` — raise `PrimitiveError` with `BEDDEL-EXEC-001` if not found
5. Call the workflow loader with the `agentId` to obtain a `WorkflowDefinition` — propagate `ParseError` if the loader fails
6. Create a new `WorkflowExecutor` using the registry from `context.metadata["registry"]` and execute the nested workflow with the resolved input
7. Return `ExecutionResult.output` from the nested workflow execution on success; raise `PrimitiveError` wrapping the error on failure
8. Support an optional `config["max_depth"]` (default: 5) to prevent infinite recursion — track depth via `context.metadata["call_depth"]` and raise `PrimitiveError` with `BEDDEL-EXEC-001` when exceeded
9. Log nested workflow invocation at DEBUG level via structured logging (agentId, depth, success/failure)
10. Unit tests achieve >80% coverage for `primitives/call_agent.py`

## Dev Notes

### Previous Story Insights (from E2.4)

Story 2.4 implemented the `output-generator` primitive — a pure data extraction primitive with no external calls. The `call-agent` primitive is significantly more complex: it loads and executes an entire nested workflow. Key patterns to reuse:

- **Error handling**: `PrimitiveError(message, code=ErrorCode.EXEC_STEP_FAILED, details={...})` for primitive-specific failures
- **Registration**: `register_builtins()` in `primitives/__init__.py` — add under the name `"call-agent"`
- **Logger**: Use `logging.getLogger("beddel.primitives.call_agent")`
- **Context metadata injection**: Same pattern as `llm`/`chat` primitives using `context.metadata["llm_provider"]` — the call-agent uses `context.metadata["workflow_loader"]` and `context.metadata["registry"]`

[Source: docs/stories/2.4.output-generator.md, src/beddel-py/src/beddel/primitives/llm.py]

### Architecture Context

The `call-agent` primitive enables workflow composition — a parent workflow can delegate to a child workflow by referencing its `agentId`. This is the mechanism for building complex multi-agent systems from simple, reusable workflow definitions.

From the high-level architecture diagram, `call-agent` is one of the 6 core primitives. It sits alongside `llm`, `chat`, `output-generator`, `guardrail`, and `tool` in the Primitives layer.

[Source: docs/architecture/high-level-architecture.md]

### YAML Config Shape (from fixture)

```yaml
- id: "delegate"
  type: "call-agent"
  config:
    agentId: "text-generator"
    input:
      messages: "$input.messages"
  result: "delegated"
```

After variable resolution, config arrives as:
```python
{
    "agentId": "text-generator",
    "input": {
        "messages": ["Hello", "World"]
    }
}
```

The primitive loads the `"text-generator"` workflow via the workflow_loader, executes it with the resolved input, and returns the nested workflow's output. The executor stores it under the `result` key (`"delegated"`) in `step_results`.

[Source: src/beddel-py/tests/fixtures/workflows/nested_agent.yaml]

### Design: Dependency Injection via Context Metadata

The call-agent primitive needs access to:
1. **workflow_loader** — A callable `(agent_id: str) -> WorkflowDefinition` that loads workflow YAML by agent ID
2. **registry** — The `PrimitiveRegistry` instance to pass to the child `WorkflowExecutor`

These are injected via `context.metadata`, following the same pattern as `llm_provider` in the llm/chat primitives. The engine factory (`create_engine()`, future story) will be responsible for wiring these into the context.

Type for workflow_loader: `Callable[[str], Awaitable[WorkflowDefinition]] | Callable[[str], WorkflowDefinition]`

For simplicity in this story, the loader is expected to be a sync callable `(str) -> WorkflowDefinition` (the parser's `parse_file` is sync). If the loader is async, the primitive should `await` it. Use duck typing: try `await` first, fall back to sync.

[Source: src/beddel-py/src/beddel/domain/ports.py, src/beddel-py/src/beddel/domain/executor.py]

### Recursion Protection

Nested workflows can call other nested workflows, creating potential infinite recursion. The primitive tracks depth via `context.metadata["call_depth"]`:

- Parent workflow starts with `call_depth = 0` (default)
- Each `call-agent` invocation increments depth by 1 in the child context
- Default `max_depth = 5` (configurable per step via `config["max_depth"]`)
- When `call_depth >= max_depth`, raise `PrimitiveError` before loading the workflow

The child executor receives a new `ExecutionContext` with `call_depth + 1` in metadata. The parent's `workflow_loader` and `registry` are forwarded to the child context so nested call-agent steps also work.

### Executor Instantiation

The primitive creates a fresh `WorkflowExecutor` for the child workflow:

```python
child_executor = WorkflowExecutor(registry=registry)
result = await child_executor.execute(workflow, input_data=nested_input)
```

Note: The child executor does NOT receive the parent's tracer or hooks in this story. Propagating observability to nested workflows is a future concern (E4 stories).

[Source: src/beddel-py/src/beddel/domain/executor.py]

### Data Models

`ExecutionContext`:
- `workflow_id: str` — Workflow identifier
- `input: dict[str, Any]` — Workflow input data
- `step_results: dict[str, Any]` — Results from previous steps
- `env: dict[str, str]` — Environment variables
- `metadata: dict[str, Any]` — Runtime metadata (e.g., `llm_provider`, `workflow_loader`, `registry`, `call_depth`)

`ExecutionResult`:
- `workflow_id: str`
- `success: bool`
- `output: Any` — The workflow's resolved output
- `step_results: dict[str, StepResult]`
- `error: str | None`
- `duration_ms: float`

`PrimitiveError(ExecutionError)` — for primitive-specific failures, uses `ErrorCode.EXEC_STEP_FAILED` (BEDDEL-EXEC-001)

`ParseError(BeddelError)` — raised by the workflow loader if the YAML is invalid

[Source: src/beddel-py/src/beddel/domain/models.py]
