# Story 2.7: Tool Primitive (Function Invocation)

## Status

Done

## Story

**As a** Python developer using Beddel,
**I want** a `tool` primitive that invokes registered Python functions by name,
**so that** I can extend workflows with custom logic, external API calls, and side effects without writing new primitives.

## Acceptance Criteria

1. Register a `tool` async primitive function that accepts `(config: dict, context: ExecutionContext)` and returns the tool function's output
2. Extract `config["name"]` to identify the target tool function — raise `PrimitiveError` with `BEDDEL-EXEC-001` when missing
3. Extract `config["args"]` as a dict of keyword arguments for the tool function (default to empty dict `{}` if not provided)
4. Retrieve a tool registry (dict mapping tool names to callables) from `context.metadata["tool_registry"]` — raise `PrimitiveError` with `BEDDEL-EXEC-001` if not found
5. Look up the tool function by name in the tool registry — raise `PrimitiveError` with `BEDDEL-EXEC-001` if the tool name is not registered
6. Invoke the tool function with the provided args — support both sync and async callables via `inspect.isawaitable()` pattern (consistent with call-agent's async loader support)
7. Return the tool function's output on success; raise `PrimitiveError` wrapping the error on failure
8. Support an optional `config["timeout"]` (default: `None` / no timeout) — if set, wrap the tool invocation in `asyncio.wait_for()` and raise `PrimitiveError` with `BEDDEL-EXEC-003` on timeout
9. Log tool invocation at DEBUG level via structured logging (tool name, args keys, success/failure, duration)
10. Unit tests achieve >80% coverage for `primitives/tool.py`

### Tool Primitive Design

The `tool` primitive is a **function invocation primitive** — it looks up a callable by name from a user-provided tool registry (stored in `context.metadata["tool_registry"]`) and invokes it with the resolved args.

This is the **basic** (P1) version. Full MCP integration is deferred to E6 (post-MVP). The basic version:
- Invokes local Python callables (sync or async)
- Uses a simple dict-based tool registry (`dict[str, Callable]`)
- Supports optional timeout via `asyncio.wait_for()`
- Does NOT handle MCP tool servers, tool discovery, or tool schemas

[Source: docs/prd/epic-list.md — E2.7 vs E6]

### YAML Config Shape

```yaml
- id: "fetch-data"
  type: "tool"
  config:
    name: "fetch_user_profile"
    args:
      user_id: "$input.userId"
  result: "profile"
```

After variable resolution, config arrives as:
```python
{
    "name": "fetch_user_profile",
    "args": {"user_id": "usr_123"},
}
```

The primitive looks up `"fetch_user_profile"` in `context.metadata["tool_registry"]`, calls it with `user_id="usr_123"`, and returns the result. The executor stores it under `"profile"` in `step_results`.

Optional timeout:
```yaml
- id: "slow-tool"
  type: "tool"
  config:
    name: "expensive_computation"
    args:
      data: "$stepResult.prepared"
    timeout: 30
  result: "computed"
```

### Data Models

The tool primitive uses minimal models from `beddel.domain.models`:

`ExecutionContext`:
- `workflow_id: str` — Workflow identifier
- `input: dict[str, Any]` — Workflow input data
- `step_results: dict[str, Any]` — Results from previous steps
- `env: dict[str, str]` — Environment variables
- `metadata: dict[str, Any]` — Runtime metadata (includes `tool_registry`)

`PrimitiveError(ExecutionError)` — for primitive-specific failures
- `ErrorCode.EXEC_STEP_FAILED` (BEDDEL-EXEC-001) — general tool errors
- `ErrorCode.EXEC_TIMEOUT` (BEDDEL-EXEC-003) — timeout errors

Note: `LLMRequest`, `LLMResponse`, `Message`, `ProviderError` are NOT needed.

### Primitive Function Signature

Type: `PrimitiveFunc = Callable[[dict[str, Any], ExecutionContext], Awaitable[Any]]`

```python
async def tool_primitive(config: dict[str, Any], context: ExecutionContext) -> Any:
    """Invoke a registered tool function by name.

    The VariableResolver has already substituted all variable references.
    This primitive looks up config["name"] in context.metadata["tool_registry"]
    and invokes it with config["args"] as keyword arguments.
    """
```

[Source: src/beddel-py/src/beddel/domain/registry.py]

### Implementation Approach

The implementation follows the call-agent pattern for async/sync dual support (~60 lines):

1. Extract `config["name"]` (required) and `config["args"]` (default `{}`)
2. Get `tool_registry` from `context.metadata["tool_registry"]`
3. Look up the tool function by name
4. Call the function with `**args`; if result is awaitable, await it
5. If `config["timeout"]` is set, wrap in `asyncio.wait_for()`
6. Return result on success; wrap exceptions in `PrimitiveError`

Key design decisions:
- **Tool registry in metadata** (not in PrimitiveRegistry) — tools are user-provided functions, not built-in primitives. The PrimitiveRegistry manages workflow step types; the tool registry manages user functions.
- **`inspect.isawaitable()` pattern** — same as call-agent's async loader support, for backward compatibility with sync tool functions.
- **`asyncio.wait_for()` for timeout** — standard library approach, raises `asyncio.TimeoutError` which we catch and wrap in `PrimitiveError(BEDDEL-EXEC-003)`.
- **Args as kwargs** — `tool_fn(**args)` allows natural Python function signatures.

[Source: pattern from src/beddel-py/src/beddel/primitives/call_agent.py]

### Consolidated Architect Findings

#### Confirmed Strengths (both reviewers agree)
1. All 10 acceptance criteria fully met with clean implementation
2. Pattern consistency with call-agent and guardrail is excellent
3. Hexagonal architecture correctly maintained — pure domain function, no adapter leakage
4. Test helper `_ctx()` is well-designed for registry presence/absence scenarios
5. Security-conscious logging (args keys only, no values)

#### Known Limitations (accepted for v0.1.0)
1. **Timeout only applies to async callables** — sync blocking functions cannot be interrupted by `asyncio.wait_for()`. This is a known limitation of the `isawaitable` pattern, consistent with call-agent. Both reviewers flagged this. Acceptable for v0.1.0; consider `asyncio.to_thread()` wrapper in future.
2. **Tool registry in `context.metadata`** — "soft" contract, risk of metadata becoming a dumping ground. Both reviewers noted this. Recommend standardizing metadata keys or introducing a `ToolRegistry` Protocol in a future story.

#### Minor Improvement Suggestions (deferred, non-blocking)
- Validate `args` is a `dict` before unpacking (GPT-5.2 suggestion) — currently relies on Python's `**` operator to raise `TypeError`, which gets wrapped in `PrimitiveError`. Acceptable but could provide a clearer error message.
- Catch `asyncio.TimeoutError` specifically rather than bare `TimeoutError` (GPT-5.2 suggestion) — in Python 3.11+, `TimeoutError` is the base class and `asyncio.TimeoutError` is an alias, so catching `TimeoutError` is correct and more general.
