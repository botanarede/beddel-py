# Story 1.3: Async Workflow Executor

## Status

Done

## Story

**As a** Python developer using Beddel,
**I want** workflows to execute their steps asynchronously in sequential order with proper error handling and lifecycle integration,
**so that** I can run multi-step AI pipelines that handle failures gracefully and support observability hooks.

## Acceptance Criteria

1. Execute workflow steps sequentially in async mode using the registered primitives
2. Resolve step configurations via VariableResolver before passing to primitives
3. Store step outputs in ExecutionContext keyed by the step's `result` field name
4. Support conditional step execution via `condition` field (resolved and evaluated as boolean)
5. Support per-step error handling with `on_error.strategy` (fail stops workflow, skip continues)
6. Invoke lifecycle hooks (on_workflow_start, on_step_start, on_step_end, on_workflow_end, on_error) at appropriate points
7. Generate OpenTelemetry spans via ITracer port for workflow and step executions
8. Implement return template resolution: if `return` defined use it, if last step has no `result` return last output, otherwise return all step_results
9. Return ExecutionResult with workflow_id, success status, output, step_results dict, error info, and duration_ms

### Data Models

- `ExecutionResult`: workflow_id, success, output, step_results (dict[str, StepResult]), error, duration_ms
- `StepResult`: step_id, output, success, error, duration_ms
- `ExecutionContext`: workflow_id (auto-generated uuid hex[:12]), input, step_results, env, metadata
- `ExecutionContext.with_step_result(key, result)` — immutable update pattern via model_copy
- `ErrorHandler`: strategy field (fail | skip)
- `ExecutionError`, `PrimitiveError` — domain exceptions re-raised directly by executor

[Source: docs/architecture/data-models.md, src/beddel-py/src/beddel/domain/models.py]

### Port Interfaces

- `ITracer` (Protocol, runtime_checkable): `start_workflow_span(workflow) -> Any`, `start_step_span(step, parent) -> Any`, `end_span(span, *, error: str | None = None) -> None`
- `ILifecycleHook` (Protocol, runtime_checkable): `on_workflow_start(workflow, input_data)`, `on_step_start(step)`, `on_step_end(step, result)`, `on_workflow_end(workflow, result)`, `on_error(error)` — all async
- `ILLMProvider` (Protocol, runtime_checkable): `complete(request)`, `stream(request)` — not used directly by executor

[Source: src/beddel-py/src/beddel/domain/ports.py, docs/architecture/components.md]

### Return Template Logic (3-way)

1. If `workflow.return_template` is defined: resolve it via `resolver.resolve_dict()` and return
2. If last step has no `result` field (is None): return `last_output` directly
3. Otherwise: return `context.step_results` (all accumulated results)

[Source: docs/architecture/data-models.md#WorkflowDefinition, executor.py _resolve_output()]

### Error Handling Flow

Step-level errors follow two paths:
- `ExecutionError` / `PrimitiveError`: hooks notified, tracer span ended with error, exception re-raised (bypasses on_error strategy)
- Generic `Exception`: formatted as error message, hooks notified, tracer span ended, then on_error strategy applied (skip → continue with failed StepResult, fail → return failed StepResult which triggers workflow failure in execute())

Workflow-level: uncaught exceptions trigger `on_error` hooks and tracer cleanup, then re-raise.
