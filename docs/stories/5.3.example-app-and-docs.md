# Story 5.3: Example Application and Documentation

## Status

InProgress

## Story

**As a** Python developer evaluating or onboarding to Beddel,
**I want** a working example FastAPI application with accompanying documentation that demonstrates the full Beddel workflow lifecycle (blocking and SSE streaming),
**so that** I can quickly understand how to integrate Beddel into my own web application, see the SDK's public API in action, and have a reference implementation to copy from.

## Acceptance Criteria

1. Create an example FastAPI application in `src/beddel-py/examples/fastapi_app.py` that exposes at least two endpoints: one blocking workflow and one SSE streaming workflow
2. Create example YAML workflow files in `src/beddel-py/examples/workflows/` — at minimum a `simple.yaml` (single-step blocking) and a `streaming.yaml` (single-step with `stream: true`)
3. The example app uses `create_beddel_handler` from `beddel.integrations.fastapi` to mount workflow handlers on FastAPI routes
4. The example app includes a health check endpoint (`GET /health`) that returns `{"status": "ok"}`
5. The example app is runnable with `uvicorn examples.fastapi_app:app` from the `src/beddel-py/` directory (after installing `beddel[fastapi]`)
6. Update `src/beddel-py/README.md` with a Quick Start section that covers: installation (`pip install beddel` and `pip install beddel[fastapi]`), a minimal code example, and instructions to run the example app
7. The README documents the public API surface: `create_beddel_handler`, `BeddelSSEAdapter`, `SSEEvent`, and key domain types (`WorkflowDefinition`, `ExecutionResult`, `WorkflowExecutor`)
8. Add inline comments in the example app explaining each section (imports, workflow loading, handler creation, route mounting)
9. Write a smoke test in `tests/integration/test_example_app.py` that imports the example app, sends a request to `/health`, and verifies the response — ensuring the example doesn't bit-rot
10. All existing tests continue to pass after changes

## Dev Notes

### Previous Story Insights (from E5.2 — SSE Streaming Response Adapter)

Story 5.2 extracted SSE logic into a framework-agnostic `BeddelSSEAdapter` in `integrations/sse.py`. Key patterns:
- `BeddelSSEAdapter` is stateless — safe for concurrent use, instantiate per-request or share
- `SSEEvent` dataclass with `serialize()` method for raw wire format
- `fastapi.py` was refactored to use `BeddelSSEAdapter` via `_sse_bridge()` — converts `SSEEvent` instances to dicts for `EventSourceResponse`
- QA tracked SSE-001 (private `_build_error_event` imported across module boundary) as medium severity — not blocking, tracked for cleanup
- `integrations/__init__.py` exports `BeddelSSEAdapter` and `SSEEvent` unconditionally, `create_beddel_handler` conditionally (requires FastAPI)

[Source: docs/stories/5.2.sse-streaming-adapter.md — Dev Notes, QA Results]

### Architecture Context

The example application demonstrates the integration layer consuming the domain core. The architecture flow is:

```
User → FastAPI → create_beddel_handler → WorkflowExecutor → Primitives → LiteLLMAdapter
```

For SSE streaming:
```
User → FastAPI → EventSourceResponse → BeddelSSEAdapter → WorkflowExecutor → AsyncIterator[str]
```

The example app is NOT part of the SDK package — it lives in `examples/` alongside the source, serving as documentation and a reference implementation.

[Source: docs/architecture/core-workflows.md, docs/architecture/components.md]

### Public API Surface (for README documentation)

```python

    ProviderError,

```

### `create_beddel_handler` Factory Signature

```python
def create_beddel_handler(
    workflow: WorkflowDefinition | str,
    *,
    provider: ILLMProvider | None = None,
    hooks: list[ILifecycleHook] | None = None,
    tracer: ITracer | None = None,
    registry: PrimitiveRegistry | None = None,
) -> Callable[..., Awaitable[Any]]:
```

Accepts a `WorkflowDefinition` or a `str` path to a YAML file. Returns an async handler that accepts a `Request` and returns `JSONResponse` (blocking) or `EventSourceResponse` (streaming).

[Source: src/beddel-py/src/beddel/integrations/fastapi.py]

### Technical Constraints

- Python 3.11+ with async/await [Source: docs/architecture/tech-stack.md]
- `from __future__ import annotations` for deferred evaluation [Source: docs/architecture/coding-standards.md]
- Type hints required on all public functions [Source: docs/architecture/coding-standards.md]
