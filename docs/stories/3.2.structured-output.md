# Story 3.2: Structured Output Enforcement (Pydantic)

## Status

Done

## Story

**As a** Python developer using Beddel,
**I want** structured output enforcement that validates LLM responses against Pydantic models,
**so that** my workflows can reliably extract typed, validated data from LLM calls without manual JSON parsing or error handling.

## Acceptance Criteria

1. Create a `StructuredOutputHandler` class in `src/beddel-py/src/beddel/adapters/structured.py` that takes a Pydantic model class and produces the `response_format` dict for LiteLLM
2. Implement `def to_response_format(self) -> dict[str, Any]` that converts a Pydantic model into a JSON Schema `response_format` parameter compatible with LiteLLM's structured output API (`{"type": "json_schema", "json_schema": {"name": ..., "strict": True, "schema": ...}}`)
3. Implement `def parse_response(self, content: str) -> BaseModel` that parses raw LLM response content (JSON string) into a validated Pydantic model instance
4. Raise `PrimitiveError` with `BEDDEL-EXEC-001` when LLM response content is not valid JSON
5. Raise `PrimitiveError` with `BEDDEL-EXEC-001` when parsed JSON does not conform to the Pydantic model schema (validation error)
6. Integrate with the `llm` primitive: when `config["response_model"]` is set, automatically apply `StructuredOutputHandler` to build `response_format` and parse/validate the response
7. The `llm` primitive returns the validated Pydantic model instance (as a dict via `.model_dump()`) instead of raw `LLMResponse` when `response_model` is configured
8. Support nested Pydantic models, `Optional` fields, `list[...]` fields, and `Literal` types in the schema generation
9. Export `StructuredOutputHandler` from `beddel.adapters`
10. Unit tests achieve >80% coverage for `adapters/structured.py` and the new `llm` primitive structured output path

### LiteLLM Structured Output Format

LiteLLM expects `response_format` in this shape for JSON Schema enforcement:

```python
{
    "type": "json_schema",
    "json_schema": {
        "name": "ModelName",       # from model.__name__
        "strict": True,
        "schema": { ... }          # from model.model_json_schema()
    }
}
```

This is passed directly to the provider (OpenAI, Gemini, etc.) which enforces the schema at generation time. The LLM response content will be a JSON string conforming to the schema.

[Source: docs/architecture/external-apis.md, https://docs.litellm.ai/]

### Pydantic JSON Schema Generation

Pydantic v2 provides `model.model_json_schema()` which generates a JSON Schema dict from a model class. This handles:
- Nested models (via `$defs` references)
- `Optional[T]` fields (anyOf with null)
- `list[T]` fields (array type with items)
- `Literal["a", "b"]` (enum constraint)
- Field descriptions, defaults, constraints

The generated schema can be passed directly as the `schema` value in the LiteLLM format.

[Source: Pydantic v2 documentation]

### How the LLM Primitive Currently Handles response_format

In `primitives/llm.py`, `_build_request()` already checks for `config["response_format"]` and passes it to `LLMRequest`. The new `response_model` config key is a higher-level abstraction: the user provides a Pydantic model class, and the handler converts it to the low-level `response_format` dict automatically.

Flow with `response_model`:
```
config["response_model"] = MyModel
  → StructuredOutputHandler(MyModel)
  → handler.to_response_format() → LLMRequest.response_format
  → LiteLLMAdapter passes to litellm.acompletion()
  → LLM returns JSON string
  → handler.parse_response(content) → MyModel instance
  → instance.model_dump() → returned as step result
```

Flow without `response_model` (unchanged):
```
config["response_format"] = {...}  (optional, raw dict)
  → LLMRequest.response_format = {...}
  → LiteLLMAdapter passes to litellm.acompletion()
  → LLMResponse returned as-is
```
