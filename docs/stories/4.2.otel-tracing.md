# Story 4.2: OpenTelemetry Span Generation

## Status

Done

## Story

**As a** Python developer using Beddel,
**I want** an `OpenTelemetryAdapter` that implements the `ITracer` protocol and generates spans for workflow and step executions,
**so that** I can observe distributed traces of my AI workflows using any OpenTelemetry-compatible backend (Jaeger, Zipkin, LangSmith via OTLP) without modifying the core executor.

## Acceptance Criteria

1. Create an `OpenTelemetryAdapter` class in `src/beddel-py/src/beddel/adapters/tracing.py` that implements the `ITracer` protocol from `domain/ports.py`
2. `start_workflow_span(workflow: WorkflowDefinition) -> Span` creates a root span named `beddel.workflow.{workflow.metadata.name}` with attributes: `beddel.workflow.name`, `beddel.workflow.version`, `beddel.workflow.id` (from context or generated)
3. `start_step_span(step: StepDefinition, parent: Span | None) -> Span` creates a child span named `beddel.step.{step.id}` with attributes: `beddel.step.id`, `beddel.step.type`, linked to the parent workflow span
4. `end_span(span: Span, *, error: str | None = None)` ends the span — if `error` is provided, records it as a span event and sets span status to `ERROR`; otherwise sets status to `OK`
5. The adapter uses `opentelemetry.trace.get_tracer("beddel")` to obtain a tracer instance — it does NOT create its own `TracerProvider` (users configure their own exporter/provider)
6. `isinstance(OpenTelemetryAdapter(), ITracer)` returns `True` (protocol compliance)
7. Export `OpenTelemetryAdapter` from `beddel.adapters`
8. Unit tests achieve >80% coverage for `adapters/tracing.py`
9. All span operations are wrapped in try/except — tracing failures MUST NOT crash the workflow (log warning and continue)
10. Span attributes follow OpenTelemetry semantic conventions where applicable, using the `beddel.*` namespace for custom attributes

### Architecture Context

The `OpenTelemetryAdapter` is one of three adapters in the hexagonal architecture. It implements the `ITracer` port and provides span generation for distributed tracing.

From the component diagram:
- `WorkflowExecutor` depends on `ITracer` (port)
- `OpenTelemetryAdapter` implements `ITracer` (adapter)
- The executor calls `start_workflow_span()`, `start_step_span()`, and `end_span()` at the appropriate lifecycle points

[Source: docs/architecture/components.md#OpenTelemetryAdapter]

### ITracer Protocol (Current State)

The protocol is defined in `domain/ports.py` with these methods:
- `start_workflow_span(workflow: WorkflowDefinition) -> Any` — returns a span handle
- `start_step_span(step: StepDefinition, parent: Any) -> Any` — returns a child span handle
- `end_span(span: Any, *, error: str | None = None) -> None` — ends span, optionally recording error

The return type is `Any` to keep the port framework-agnostic. The adapter will return `opentelemetry.trace.Span` objects.

[Source: src/beddel-py/src/beddel/domain/ports.py]

### WorkflowExecutor Tracer Integration (Current State)

The executor already dispatches tracer calls at these points:
- `start_workflow_span(workflow)` — before step loop, stores as `workflow_span`
- `start_step_span(step, parent_span)` — before each step execution (after condition check)
- `end_span(step_span)` — after successful step execution
- `end_span(step_span, error=str(exc))` — on step error
- `end_span(workflow_span)` — after successful workflow completion
- `end_span(workflow_span, error=str(exc))` — on workflow error

The executor accepts `tracer: ITracer | None = None` in its constructor. All tracer calls are guarded with `if self._tracer:`.

[Source: src/beddel-py/src/beddel/domain/executor.py]

### OpenTelemetry API Patterns

Key API usage for the adapter:
```python
from opentelemetry import trace
from opentelemetry.trace import StatusCode

tracer = trace.get_tracer("beddel")

# Root span
span = tracer.start_span("beddel.workflow.my-agent")
span.set_attribute("beddel.workflow.name", "my-agent")

# Child span (using parent context)
ctx = trace.set_span_in_context(parent_span)
child = tracer.start_span("beddel.step.step-1", context=ctx)

# End with error
span.set_status(StatusCode.ERROR, "error message")
span.record_exception(exception)
span.end()

# End success
span.set_status(StatusCode.OK)
span.end()
```

### Security Review
No security concerns. The adapter is an internal SDK component — it only reads workflow/step metadata and writes span attributes. No external input surfaces, no credential handling. All operations are synchronous (OTel span API is sync by design), so no event loop blocking. The adapter adds negligible overhead to workflow execution.
