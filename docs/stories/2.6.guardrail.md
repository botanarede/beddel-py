# Story 2.6: Guardrail Primitive (Input/Output Validation)

## Status

Done

## Story

**As a** Python developer using Beddel,
**I want** a `guardrail` primitive that validates data against a JSON Schema definition,
**so that** I can enforce structural and type constraints on workflow step inputs and outputs, catching invalid data early before it propagates through the pipeline.

## Acceptance Criteria

1. Register a `guardrail` async primitive function that accepts `(config: dict, context: ExecutionContext)` and returns the validated data on success
2. Extract `config["schema"]` as the JSON Schema definition — raise `PrimitiveError` with `BEDDEL-EXEC-001` when missing
3. Extract `config["input"]` as the data to validate (already resolved by VariableResolver) — raise `PrimitiveError` with `BEDDEL-EXEC-001` when missing
4. Validate the input data against the JSON Schema using the `jsonschema` library — on success, return the input data as-is (passthrough pattern)
5. On validation failure with default behavior: raise `PrimitiveError` with `BEDDEL-EXEC-001` including validation error details (message, path, schema_path)
6. Support optional `config["on_fail"]` strategy: `"raise"` (default) raises `PrimitiveError`; `"return_errors"` returns a dict with `{"valid": False, "errors": [...]}` instead of raising
7. On success with `on_fail` set to `"return_errors"`: return `{"valid": True, "data": <input>}` for consistency
8. Log validation attempts at DEBUG level via structured logging (schema type, input type, result)
9. Add `jsonschema>=4.23` to project dependencies in `pyproject.toml`
10. Unit tests achieve >80% coverage for `primitives/guardrail.py`

### Guardrail vs Other Primitives — Key Difference

The `llm` and `chat` primitives call an external LLM provider. The `output-generator` extracts a template. The `call-agent` invokes nested workflows. The `guardrail` is a **pure validation primitive** — it takes the already-resolved config, validates `config["input"]` against `config["schema"]` using JSON Schema, and either returns the data or raises an error.

The VariableResolver has already substituted all `$stepResult.*` and `$input.*` references before the primitive receives the config. The guardrail's job is to validate the resolved data against the schema definition.

### YAML Config Shape (from fixtures)

```yaml
- id: "validate"
  type: "guardrail"
  config:
    schema:
      type: "object"
      properties:
        text:
          type: "string"
          minLength: 1
    input: "$stepResult.delegated"
  result: "validated"
```

After variable resolution, config arrives as:
```python
{
    "schema": {
        "type": "object",
        "properties": {
            "text": {
                "type": "string",
                "minLength": 1,
            },
        },
    },
    "input": {"text": "Hello, world!"},  # resolved from $stepResult.delegated
}
```

The primitive validates `config["input"]` against `config["schema"]`. On success, returns the input dict as-is. The executor stores it under the `result` key (`"validated"`) in `step_results`.

Optional `on_fail` strategy:
```yaml
- id: "validate"
  type: "guardrail"
  config:
    schema:
      type: "object"
      properties:
        text:
          type: "string"
    input: "$stepResult.delegated"
    on_fail: "return_errors"
  result: "validation_result"
```

With `on_fail: "return_errors"`, instead of raising on failure, returns:
```python
{"valid": False, "errors": [{"message": "...", "path": "...", "schema_path": "..."}]}
```

On success with `on_fail: "return_errors"`, returns:
```python
{"valid": True, "data": {"text": "Hello, world!"}}
```

### Implementation Approach

The implementation uses `jsonschema.validate()` for JSON Schema validation (~40 lines):

```python
from jsonschema import validate, ValidationError as JsonSchemaValidationError

async def guardrail_primitive(config: dict[str, Any], context: ExecutionContext) -> Any:
    if "schema" not in config:
        raise PrimitiveError(
            "guardrail requires a 'schema' key in config",
            code=ErrorCode.EXEC_STEP_FAILED,
            details={"primitive": "guardrail", "hint": "Add schema field to config"},
        )
    if "input" not in config:
        raise PrimitiveError(
            "guardrail requires an 'input' key in config",
            code=ErrorCode.EXEC_STEP_FAILED,
            details={"primitive": "guardrail", "hint": "Add input field to config"},
        )

    schema = config["schema"]
    input_data = config["input"]
    on_fail = config.get("on_fail", "raise")

    try:
        validate(instance=input_data, schema=schema)
    except JsonSchemaValidationError as exc:
        error_detail = {
            "message": exc.message,
            "path": list(exc.absolute_path),
            "schema_path": list(exc.absolute_schema_path),
        }
        if on_fail == "return_errors":
            return {"valid": False, "errors": [error_detail]}
        raise PrimitiveError(
            f"Guardrail validation failed: {exc.message}",
            code=ErrorCode.EXEC_STEP_FAILED,
            details={"primitive": "guardrail", **error_detail},
        ) from exc

    logger.debug("Guardrail validation passed: schema_type=%s", schema.get("type", "unknown"))

    if on_fail == "return_errors":
        return {"valid": True, "data": input_data}
    return input_data
```

Key design decisions:
- **`jsonschema` library** (not Pydantic) for validation — the fixture uses JSON Schema format (`type`, `properties`, `minLength`), which is the native format for `jsonschema`
- **Passthrough on success** — returns input data as-is (like `output-generator`)
- **`on_fail` strategy** — `"raise"` (default) or `"return_errors"` for non-throwing validation
- **Error details** — includes `message`, `path` (deque → list), `schema_path` (deque → list) for debugging
- **Import alias** — `from jsonschema import ValidationError as JsonSchemaValidationError` to avoid confusion with Pydantic's `ValidationError`
