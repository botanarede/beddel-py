# Story 2.1: LLM Primitive (Single-Turn, Blocking)

## Status

InProgress

## Story

**As a** Python developer using Beddel,
**I want** an `llm` primitive that executes single-turn LLM calls through the ILLMProvider port,
**so that** I can define workflow steps that invoke language models with system prompts, user messages, and optional structured output schemas.

## Acceptance Criteria

1. Register an `llm` async primitive function that accepts `(config: dict, context: ExecutionContext)` and returns an `LLMResponse`
2. Build an `LLMRequest` from the resolved config: extract `model`, `messages`, `temperature`, `max_tokens`, `response_format` fields
3. Support a `system` shorthand in config that prepends a system message to the messages list
4. Support a `provider` field in config for future routing (stored but not used in this story â€” LiteLLM adapter handles routing)
5. Obtain the `ILLMProvider` instance from the `ExecutionContext.metadata["llm_provider"]` key
6. Call `provider.complete(request)` and return the `LLMResponse`
7. Raise `PrimitiveError` with `BEDDEL-EXEC-001` when `llm_provider` is missing from context metadata
8. Raise `ProviderError` with `BEDDEL-PROVIDER-001` when the provider call fails, wrapping the original exception
9. Unit tests achieve >80% coverage for `primitives/llm.py`

### Data Models

The llm primitive works with these existing models from `beddel.domain.models`:

`LLMRequest`:
- `model: str` â€” Model identifier (e.g., "openrouter/anthropic/claude-3")
- `messages: list[Message]` â€” Conversation messages
- `temperature: float = 0.7` â€” Sampling temperature
- `max_tokens: int | None = None` â€” Maximum response tokens
- `response_format: dict[str, Any] | None = None` â€” Structured output schema
- `stream: bool = False` â€” Streaming flag (not used in this story)

`LLMResponse`:
- `content: str` â€” Response content
- `model: str` â€” Actual model used
- `usage: TokenUsage` â€” Token consumption (prompt_tokens, completion_tokens, total_tokens)
- `finish_reason: str = "stop"` â€” Completion reason
- `created_at: datetime` â€” Timestamp

`Message`:
- `role: str` â€” system | user | assistant
- `content: str`

`PrimitiveError(ExecutionError)` â€” for primitive-specific failures, uses `ErrorCode.EXEC_STEP_FAILED` (BEDDEL-EXEC-001)
`ProviderError(ExecutionError)` â€” for LLM provider failures, uses `ErrorCode.PROVIDER_ERROR` (BEDDEL-PROVIDER-001)

[Source: src/beddel-py/src/beddel/domain/models.py]

### ILLMProvider Port Interface

```python
@runtime_checkable
class ILLMProvider(Protocol):
    async def complete(self, request: LLMRequest) -> LLMResponse: ...
    async def stream(self, request: LLMRequest) -> AsyncIterator[str]: ...
```

The llm primitive only uses `complete()` in this story. Streaming is E2.2.

[Source: src/beddel-py/src/beddel/domain/ports.py]

### Primitive Function Signature

The executor calls primitives as: `await primitive_fn(resolved_config, context)`

Type: `PrimitiveFunc = Callable[[dict[str, Any], ExecutionContext], Awaitable[Any]]`

The config dict is already resolved by the VariableResolver before reaching the primitive. The primitive receives final values, not `$input.*` references.

[Source: src/beddel-py/src/beddel/domain/registry.py, executor.py]

### Provider Injection Pattern

The `ILLMProvider` instance is injected into `ExecutionContext.metadata["llm_provider"]` by the engine setup code (not yet implemented â€” will be in `create_engine()` factory). For this story, tests mock the provider and inject it into context metadata.

[Source: docs/architecture/components.md#llm Primitive]
