# Story 4.1: Granular Lifecycle Hooks

## Status

Done

## Story

**As a** Python developer using Beddel,
**I want** a `LifecycleHooksAdapter` that manages lifecycle hook registration and dispatches events (`on_step_start`, `on_step_end`, `on_llm_start`, `on_llm_end`, `onFinish`, `onError`) to registered callbacks,
**so that** I can observe and react to workflow execution events for logging, monitoring, and custom integrations without modifying the core executor.

## Acceptance Criteria

1. Create a `LifecycleHooksAdapter` class in `src/beddel-py/src/beddel/adapters/hooks.py` that implements the `ILifecycleHook` protocol and manages multiple registered callbacks
2. `LifecycleHooksAdapter` provides `register(event: str, callback: Callable) -> None` to subscribe callbacks to specific lifecycle events
3. `LifecycleHooksAdapter` provides `emit(event: str, **kwargs) -> None` (async) that dispatches events to all registered callbacks for that event, swallowing callback exceptions with a warning log (never crashing the workflow)
4. Support the following lifecycle events: `on_workflow_start`, `on_step_start`, `on_step_end`, `on_workflow_end`, `on_error` (matching existing `ILifecycleHook` protocol), plus `on_llm_start` and `on_llm_end` (FR15)
5. Extend the `ILifecycleHook` protocol in `domain/ports.py` with `on_llm_start(request: LLMRequest) -> None` and `on_llm_end(request: LLMRequest, response: LLMResponse) -> None` methods
6. Integrate `on_llm_start` / `on_llm_end` hook calls into the `llm` primitive (`primitives/llm.py`) ‚Äî fire `on_llm_start` before calling the adapter and `on_llm_end` after receiving the response
7. `LifecycleHooksAdapter` supports both sync and async callbacks (auto-detects and wraps sync callbacks in `asyncio.to_thread` or calls directly)
8. Export `LifecycleHooksAdapter` from `beddel.adapters`
9. Unit tests achieve >80% coverage for `adapters/hooks.py` and the modified code paths in `primitives/llm.py`
10. All callback exceptions are caught and logged as warnings ‚Äî they MUST NOT propagate to the workflow executor or interrupt execution

### Architecture Context

The `LifecycleHooksAdapter` is one of three adapters in the hexagonal architecture. It implements the `ILifecycleHook` port and provides an event emitter pattern for dispatching lifecycle events.

From the component diagram:
- `WorkflowExecutor` depends on `ILifecycleHook` (port)
- `LifecycleHooksAdapter` implements `ILifecycleHook` (adapter)
- The adapter manages registration and dispatch ‚Äî the executor just calls protocol methods

[Source: docs/architecture/components.md#LifecycleHooksAdapter]

### ILifecycleHook Protocol (Current State)

The protocol is defined in `domain/ports.py` with these methods:
- `on_workflow_start(workflow: WorkflowDefinition, input_data: dict[str, Any]) -> None`
- `on_step_start(step: StepDefinition) -> None`
- `on_step_end(step: StepDefinition, result: Any) -> None`
- `on_workflow_end(workflow: WorkflowDefinition, result: Any) -> None`
- `on_error(error: Exception) -> None`

This story adds:
- `on_llm_start(request: LLMRequest) -> None` (FR15)
- `on_llm_end(request: LLMRequest, response: LLMResponse) -> None` (FR15)

[Source: src/beddel-py/src/beddel/domain/ports.py]

### WorkflowExecutor Hook Integration (Current State)

The executor already dispatches hooks via `for hook in self._hooks: await hook.method()` at these points:
- `on_workflow_start` ‚Äî before step loop
- `on_step_start` ‚Äî before each step execution (after condition check)
- `on_step_end` ‚Äî after successful step execution
- `on_workflow_end` ‚Äî after successful workflow completion
- `on_error` ‚Äî on any exception (step-level and workflow-level)

The executor accepts `hooks: list[ILifecycleHook] | None = None` in its constructor. The `LifecycleHooksAdapter` will be passed as a single hook that internally manages multiple callbacks.

[Source: src/beddel-py/src/beddel/domain/executor.py]

### on_error Hook Firing Rules

| Exception Type | Step-level `on_error` | Workflow-level `on_error` | Behavior |
|---|---|---|---|
| `ExecutionError` / `PrimitiveError` | ‚úÖ fires | ‚úÖ fires (re-raised) | Domain errors propagate up after notifying hooks |
| Generic `Exception` | ‚úÖ fires | ‚ùå (handled at step) | Converted to `StepResult(success=False)` |

[Source: docs/architecture/error-handling-strategy.md]

### FR15-FR17 Requirements

- **FR15:** Provide granular lifecycle hooks: `on_step_start`, `on_step_end`, `on_llm_start`, `on_llm_end`
- **FR16:** Execute `onFinish` callback with complete workflow result (maps to `on_workflow_end`)

Two architect reviews completed (Winston APPROVE, GPT-5.2 CONCERNS). GPT-5.2 raised 6 concerns ‚Äî 3 actionable (A, B, E), 3 informational (C, D, F). After analysis, concern A (`asyncio.iscoroutinefunction` edge cases with callable objects and functools.partial) is the most substantive ‚Äî it could silently call an async callable synchronously in exotic scenarios. Concern B (`callback.__name__` on lambdas/partials) only affects DEBUG logging and is low risk. ### Improvements Checklist
- [ ] ASYNC-001: Harden async detection ‚Äî replace `asyncio.iscoroutinefunction(callback)` with try/inspect.isawaitable pattern to handle callable objects with async `__call__` and functools.partial wrapping coroutines (GPT-5.2 concern A)
- [ ] LOG-001: Use safe callback name resolution ‚Äî replace `callback.__name__` in register() DEBUG log with `getattr(callback, "__name__", repr(callback))` to handle lambdas, partials, and callable objects (GPT-5.2 concern B)
- [ ] SAFE-002: Add list snapshot in emit() ‚Äî iterate over `list(self._listeners.get(event, []))` instead of the live list to prevent mutation during iteration (GPT-5.2 concern E)

### Security Review
No security concerns. The adapter is an internal SDK component ‚Äî callbacks are developer-registered, not user-supplied. The `lifecycle_hooks` metadata key uses the same stringly-typed pattern as `llm_provider`, consistent with existing conventions. No external input surfaces, no credential handling, no network calls.

### Performance Considerations
Sequential callback dispatch in `emit()` means a slow callback delays subsequent callbacks and the workflow. This is acceptable for v0.1.0 ‚Äî lifecycle hooks are intended for lightweight observability (logging, metrics), not heavy computation. GPT-5.2 suggested optional parallel mode (concern C) as a future enhancement. The streaming path correctly skips `on_llm_end` since no materialized `LLMResponse` exists ‚Äî this avoids blocking the stream for hook dispatch.
