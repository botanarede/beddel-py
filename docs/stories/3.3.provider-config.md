# Story 3.3: Provider-Specific Configuration (OpenRouter, Gemini, Bedrock)

## Status

Done

## Story

**As a** Python developer using Beddel,
**I want** provider-specific configuration support that maps provider names to LiteLLM model prefixes and default parameters,
**so that** my workflows can target OpenRouter, Google Gemini, and AWS Bedrock without manually constructing LiteLLM model strings or passing provider-specific kwargs every time.

## Acceptance Criteria

1. Create a `ProviderConfig` Pydantic model in `src/beddel-py/src/beddel/adapters/provider_config.py` that holds provider-specific settings: `provider` name, `api_key` (optional), `api_base` (optional), `model_prefix` (optional), and `extra_params` (optional dict for provider-specific kwargs)
2. Create a `ProviderRegistry` class in the same module that maps provider names (e.g. `"openrouter"`, `"gemini"`, `"bedrock"`) to their `ProviderConfig` with sensible defaults for `model_prefix` (e.g. `"openrouter/"` for OpenRouter, `"gemini/"` for Gemini, `"bedrock/"` for Bedrock)
3. `ProviderRegistry` provides `register(name, config)` to add/override providers and `get(name)` to retrieve a provider config, raising `ConfigurationError` with `BEDDEL-CONFIG-001` when the provider name is not found
4. `ProviderRegistry` provides a `resolve_model(provider_name, model_name)` method that prepends the provider's `model_prefix` to the model name if the model name doesn't already include it (e.g. `resolve_model("openrouter", "anthropic/claude-3.5-sonnet")` → `"openrouter/anthropic/claude-3.5-sonnet"`)
5. `ProviderRegistry` provides a `create_adapter(provider_name)` factory method that returns a `LiteLLMAdapter` instance configured with the provider's `api_key` and `api_base`
6. Integrate with the `LiteLLMAdapter`: extend `_build_params()` to merge `extra_params` from `ProviderConfig` when an adapter is created via the registry (extra_params are passed as additional kwargs to `litellm.acompletion()`)
7. Provide built-in defaults for three providers: `openrouter` (prefix `"openrouter/"`), `gemini` (prefix `"gemini/"`), `bedrock` (prefix `"bedrock/"`)
8. Export `ProviderConfig` and `ProviderRegistry` from `beddel.adapters`
9. Unit tests achieve >80% coverage for `adapters/provider_config.py` and the modified `LiteLLMAdapter` code paths
10. All API keys MUST come from environment variables or explicit constructor args — never hardcoded

### LiteLLM Provider Model Prefixes

LiteLLM uses model string prefixes to route to the correct provider:

| Provider | Prefix | Example Model String |
|----------|--------|---------------------|
| OpenRouter | `openrouter/` | `openrouter/anthropic/claude-3.5-sonnet` |
| Google Gemini | `gemini/` | `gemini/gemini-2.0-flash` |
| AWS Bedrock | `bedrock/` | `bedrock/anthropic.claude-3-5-sonnet-20241022-v2:0` |
| OpenAI (default) | _(none)_ | `gpt-4o` |

Provider-specific kwargs passed to `litellm.acompletion()`:
- OpenRouter: `api_base="https://openrouter.ai/api/v1"` (set via `api_base`)
- Gemini: No special kwargs needed beyond prefix
- Bedrock: `aws_region_name`, `aws_access_key_id`, `aws_secret_access_key` (via extra_params or env vars)

[Source: docs/architecture/external-apis.md, https://docs.litellm.ai/]

### Data Models Used

`ConfigurationError(BeddelError)` — for configuration failures:
- `ErrorCode.CONFIG_INVALID` (`BEDDEL-CONFIG-001`)

`LiteLLMAdapter` — current constructor signature:
```python
def __init__(self, *, api_key: str | None = None, api_base: str | None = None) -> None
```

This story extends it to:
```python
def __init__(self, *, api_key: str | None = None, api_base: str | None = None, extra_params: dict[str, Any] | None = None) -> None
```

**Overall: Strong.** Implementation is clean, consistent, and follows established patterns from Story 3.2. The `ProviderConfig` Pydantic model is well-defined with sensible defaults. `ProviderRegistry` is a focused class with clear single responsibility — mapping provider names to configurations and producing configured adapters. The factory method `create_adapter()` correctly wires `api_key`, `api_base`, and `extra_params` through to `LiteLLMAdapter`.
