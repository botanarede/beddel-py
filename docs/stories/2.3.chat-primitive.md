# Story 2.3: Chat Primitive (Multi-Turn Conversations)

## Status

Review

## Story

**As a** Python developer using Beddel,
**I want** a `chat` primitive that manages multi-turn conversational interactions through the ILLMProvider port,
**so that** I can define workflow steps that maintain conversation history across turns, appending new user messages and collecting assistant responses.

## Acceptance Criteria

1. Register a `chat` async primitive function that accepts `(config: dict, context: ExecutionContext)` and returns an `LLMResponse`
2. Build an `LLMRequest` from the resolved config: extract `model`, `messages`, `temperature`, `max_tokens`, `response_format` fields (reuse `_build_request` pattern from llm primitive or create chat-specific builder)
3. Support a `system` shorthand in config that prepends a system message to the messages list
4. Support a `history` field in config that references a previous step result key (e.g., `"$stepResult.prev_chat"`) — when present, prepend the historical messages before the new messages
5. The `history` field value is a list of `Message`-compatible dicts (role + content) already resolved by the VariableResolver — the chat primitive receives them as plain dicts and converts to `Message` instances
6. Obtain the `ILLMProvider` instance from `ExecutionContext.metadata["llm_provider"]`
7. Call `provider.complete(request)` and return the `LLMResponse`
8. Raise `PrimitiveError` with `BEDDEL-EXEC-001` when `llm_provider` is missing from context metadata
9. Raise `ProviderError` with `BEDDEL-PROVIDER-001` when the provider call fails, wrapping the original exception
10. Unit tests achieve >80% coverage for `primitives/chat.py`

### Previous Story Insights (from E2.1, E2.2)

Stories 2.1 and 2.2 implemented the `llm` primitive with blocking and streaming modes. The `chat` primitive follows the same pattern but adds conversation history management:

- **`llm.py`** pattern: `_build_request(config)` → `LLMRequest`, then `provider.complete(request)` or `provider.stream(request)`. The chat primitive only needs blocking mode (no streaming in this story).
- **Error handling**: Same pattern — `PrimitiveError(BEDDEL-EXEC-001)` for missing provider, `ProviderError(BEDDEL-PROVIDER-001)` for provider failures with `raise ... from exc` chaining.
- **Registration**: `register_builtins()` in `primitives/__init__.py` — add `chat_primitive` alongside `llm_primitive`.
- **Logger**: Use `logging.getLogger("beddel.primitives.chat")`.

[Source: src/beddel-py/src/beddel/primitives/llm.py, docs/stories/2.1.llm-primitive.md, docs/stories/2.2.llm-streaming.md]

### Chat vs LLM Primitive — Key Difference

The `llm` primitive is single-turn: it takes messages from config and sends them to the provider. The `chat` primitive is multi-turn: it can incorporate message history from a previous step result.

Message assembly order:
1. System message (from `system` shorthand, if present)
2. History messages (from `history` field, if present — already resolved by VariableResolver)
3. New messages (from `messages` field in config)

This allows building conversational flows:
```yaml
- id: "first-turn"
  type: "chat"
  config:
    model: "openrouter/anthropic/claude-3"
    system: "You are a helpful assistant."
    messages:
      - role: "user"
        content: "$input.question"
  result: "turn1"

- id: "second-turn"
  type: "chat"
  config:
    model: "openrouter/anthropic/claude-3"
    system: "You are a helpful assistant."
    history:
      - role: "user"
        content: "$input.question"
      - role: "assistant"
        content: "$stepResult.turn1.content"
    messages:
      - role: "user"
        content: "Can you elaborate on that?"
  result: "turn2"
```

After variable resolution, the `history` field arrives as a list of plain dicts:
```python
[
    {"role": "user", "content": "What is Python?"},
    {"role": "assistant", "content": "Python is a programming language..."},
]
```

The chat primitive converts these to `Message` instances and inserts them in the correct position.

[Source: docs/architecture/components.md, docs/prd/requirements.md FR6]

### Data Models

The chat primitive uses the same models as the llm primitive from `beddel.domain.models`:

`LLMRequest`:
- `model: str` — Model identifier
- `messages: list[Message]` — Conversation messages (system + history + new)
- `temperature: float = 0.7`
- `max_tokens: int | None = None`
- `response_format: dict[str, Any] | None = None`
- `stream: bool = False` — Not used in this story (chat is blocking only)

`LLMResponse`:
- `content: str` — Response content
- `model: str` — Actual model used
- `usage: TokenUsage` — Token consumption
- `finish_reason: str = "stop"`
- `created_at: datetime`

`Message`:
- `role: str` — system | user | assistant
- `content: str`

`PrimitiveError(ExecutionError)` — `ErrorCode.EXEC_STEP_FAILED` (BEDDEL-EXEC-001)
`ProviderError(ExecutionError)` — `ErrorCode.PROVIDER_ERROR` (BEDDEL-PROVIDER-001)

[Source: src/beddel-py/src/beddel/domain/models.py]

### ILLMProvider Port Interface

```python
@runtime_checkable
class ILLMProvider(Protocol):
    async def complete(self, request: LLMRequest) -> LLMResponse: ...
    async def stream(self, request: LLMRequest) -> AsyncIterator[str]: ...
```

The chat primitive only uses `complete()` in this story.

[Source: src/beddel-py/src/beddel/domain/ports.py]

### Primitive Function Signature

Type: `PrimitiveFunc = Callable[[dict[str, Any], ExecutionContext], Awaitable[Any]]`

The config dict is already resolved by the VariableResolver before reaching the primitive. The `history` field arrives as a list of plain dicts (not `$stepResult.*` references).

[Source: src/beddel-py/src/beddel/domain/registry.py, executor.py]
